How to use these vectors

Every hash below is produced by the reference construction in VRSLib (Specification §6); the signatures are produced with the published test keys below. An implementation claiming any document-level conformance class must reproduce these values exactly. Regenerate them with Reference/04-reference-contracts/tools/generate-vectors.mjs (ethers v6).

8. Negative test cases

Starting from the valid §4 signed proof, each mutation must cause verification (Specification §9.3) to reject at the indicated step.

MutationRequired outcome
Flip one byte of productId (keep paths)reject — recomputed itemLeaf no longer folds to receiptRoot (step 3)
Flip one byte of saltCommitreject — root mismatch (step 3); also changes entitlementId
Replace issuerSig with any other valid-form signaturereject — recovered signer ≠ issuer (step 4)
Raise s above the secp256k1 half-order (malleate)reject — malleability guard (§6.4)
Use the §5 delegated receipt with issuedAt outside [validFrom, validUntil]reject — delegation window (step 5)
Present the §5 receipt to a verifier whose role ≠ merchantreject — delegate-role mismatch (step 6 / §8)
Supply headerPath.length ≠ itemPath.length, or length > 16reject — path-length / depth check (step 1)
Swap order of a node() pair (wrong index bit)reject — folds to a different root (step 2/3)
Register with the §3 proof (which carries saltCommit, not salt)reject — bearer authority not established; the derived id is inert (§14)
Re-use a consumed serial nullifierreject — set-once nullifier already used (§16)
