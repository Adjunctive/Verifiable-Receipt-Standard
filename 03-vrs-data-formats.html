VRS 1.0 — Companion to the Specification

VRS Data Formats

Canonical JSON schemas for every VRS document, with worked examples

Introduction

This document specifies the on-the-wire JSON for every VRS document type referenced by the VRS Specification. The specification pins the cryptographic commitments (the hash inputs in §6–§7); this document pins the serialized documents that carry those values and the surrounding metadata. The two together let independent implementations produce byte-identical artifacts.

What is committed, and what is not The receipt root commits to only the header-leaf fields (issuer, issuedAt, receiptType, fiscalHash, delegationHash) and the per-line group hashes (core, commercial, warranty, descriptive, provenance). Everything else in a receipt document — receiptId, payment, payer, totals, free-text descriptive contents, and any fiscal payload — is committed indirectly (via fiscalHash, descriptiveHash, or termsHash) or is purely informational routing data. A verifier must treat unverifiable informational fields as untrusted hints.

Conventions

  • JSON is canonicalised with JCS (RFC 8785) wherever a hash is taken over it.
  • Integers used in hashing or protocol state are decimal strings; binary values are lowercase 0x hex.
  • ABSENT denotes the absent-sentinel hash (spec §6.1); a JSON null for an optional group maps to ABSENT in the commitment.
  • Column R: ● = required, ○ = optional.

2. Bearer secrets — vrs/secrets/1.0

Travels separately from the receipt and must not appear in documents, refs, or ordinary proofs. Each entry reveals a preimage whose commitment is in the receipt or provenance group.

{
  "schema": "vrs/secrets/1.0",
  "receiptId": "order-100045",
  "secrets": [
    { "lineIndex": "0", "salt": "0x63968191ce601b875cf1255410336c015731dd795bd34bdb4e56f5a633ec939f" },
    { "lineIndex": "1", "salt": "0x1aef9141b419aa03af6f5331cd870c33c18e13183a9b7113d51dff93f58c9759" },
    { "batchId": "0x33bdafc4…ab29679f", "serial": "SER1234",
      "serialSalt": "0x67854b809ea195362b018634ce712ab34c676bcadda791b450aa825c9cc7e742" }
  ]
}

3. Proof — vrs/proof/1.0

A single-line selective-disclosure proof. disclosureLevel is one of full / core / core+<group> / zk (§9.1). Disclosed groups supply preimage fields; undisclosed groups supply their group hash (or ABSENT). headerPath and itemPath are equal-length sibling arrays processed per §6.5.

{
  "schema": "vrs/proof/1.0",
  "disclosureLevel": "core+commercial",
  "header": { "issuer": "0x…00A1", "issuedAt": "1790000000", "receiptType": "sale",
              "fiscalHash": "0x8c58af25…8180498a", "delegationHash": "ABSENT",
              "issuerSig": "0x<65-byte>", "headerPath": ["0x", "0x"],
              "rootIdentity": "0x0000…0000", "delegRole": "0x00..00",
              "delegValidFrom": "0", "delegValidUntil": "0", "rootSig": "0x" },
  "line": { "lineIndex": "1", "core": { "productId": "(01)09506000134352(21)SER1234",
              "qtyMilli": "1000", "saltCommit": "0xf305f1fa…deb491a1" },
            "commercial": { "unitPriceMinor": "4999", "taxRateBps": "1900", "currency": "EUR" },
            "warrantyHash": "0x753fda7c…faae0961", "descriptiveHash": "0x4259954c…bb826364",
            "provenanceHash": "0xb2dbb672…bbf61c33", "itemPath": ["0x", "0x"] },
  "receiptRoot": "0x4881046b…ff8338e9"
}

4. Presentation and holder challenge

A presentation bundles a proof, a registry read, and (for a registered entitlement) a holder challenge. The challenge is signed by the current holder over an EIP-712 typed structure; the recommended type is:

HolderChallenge(bytes32 entitlementId,bytes32 nonce,string audience,uint64 expiresAt)
// signed by holderOf(entitlementId); verified by EOA recovery or ERC-1271.

{
  "schema": "vrs/presentation/warranty/1.0",
  "proof": { "disclosureLevel": "core+warranty", "lineIndex": "1", "groups": ["core","warranty"] },
  "holderChallenge": { "entitlementId": "0x47cc2d0c…b8cb979d", "nonce": "0x9f2c4a…",
                       "audience": "warranty", "expiresAt": "1790200300",
                       "holderSig": "0x" },
  "registryRead": { "status": "Active", "holder": "0x3333…3333" }
}

The vrs/presentation/resale/1.0 variant adds lifecycle[] (selected attestation hashes + documents) and a transferAuth object (§14 registry digests).

5. Credit document

A credit is a full vrs/receipt/1.0 of receiptType: credit that references its origin via headerExtensions["vrs.credit"].originRoot. Credit lines derive no entitlement; their saltCommit is the sentinel string "CREDIT_SENTINEL".

{
  "schema": "vrs/receipt/1.0", "receiptType": "credit", "receiptId": "credit-200007",
  "issuer": { "address": "0x…00A1" }, "issuedAt": "1790100000",
  "headerExtensions": { "vrs.credit": { "originRoot": "0x4881046b…ff8338e9", "originReceiptId": "order-100045" } },
  "lineItems": [ { "core": { "productId": "(01)09506000134352(21)SER1234", "qtyMilli": "1000",
      "saltCommit": "CREDIT_SENTINEL" },
      "descriptive": { "lineExtensions": { "vrs.credit": { "originLineIndex": "1",
        "creditedQtyMilli": "1000", "creditedAmountMinor": "4999", "reason": "return" } } } } ],
  "signature": { "signer": "0x…00A1", "signature": "0x<65-byte>" }
}

6. Credit graph — vrs/credit-graph/1.0

completeness must be one of complete / partial / paginated / unavailable / account-gated / retention-expired (§12).

{
  "schema": "vrs/credit-graph/1.0",
  "originRoot": "0x4881046b…ff8338e9",
  "completeness": "complete",
  "credits": [ { "receiptId": "credit-200007", "originLineIndex": "1",
                 "creditedAmountMinor": "4999", "reason": "return",
                 "documentRef": "vrs1:ref:https://receipts.example/ref/credit-200007" } ]
}

7. Lifecycle attestation — vrs/lifecycle-attestation/1.0

Bound to an entitlement; only attestationHash = keccak256(JCS(attestation)) is anchored on chain (§15). Contents stay off chain.

{
  "schema": "vrs/lifecycle-attestation/1.0",
  "entitlementId": "0x47cc2d0c…b8cb979d",
  "eventType": "service",                 // service|repair|inspection|owner-note|recall|appraisal
  "issuer": "0x6666…6666",                // attester key
  "issuedAt": "1790200000",
  "summaryHash": "0x",
  "attachments": [ { "type": "service-report", "hash": "0x" } ],
  "attestationHash": "0x",
  "signature": "0x"
}

8. Recall notice — vrs/recall-notice/1.0

Keyed on a manufacturing batchId, not on a holder (§17). The notice hash is anchored via NoticeRegistry.recall(batchId, noticeHash, severity).

{
  "schema": "vrs/recall-notice/1.0",
  "recallId": "recall-7",
  "issuer": { "address": "0x5555…5555" },     // manufacturer/authority root, attested for the batch
  "batchId": "0x33bdafc4…ab29679f",
  "scope": "batch",                            // batch | serial-range | serial-list
  "productClass": "ACME Toaster",
  "severity": "safety",                        // safety | advisory | informational
  "action": "stop-use",                        // stop-use | return | repair | update
  "remedyHash": "0x",
  "remedyURI": "https://acme.example/recall/7",
  "issuedAt": "1790500000",
  "signature": "0x"
}

9. Grant chain — vrs/grant-chain/1.0

An ordered chain of signed grants from a manufacturer root to the receipt issuer, establishing an authorised channel (§16). Each grant's parentGrant references the previous grant's hash (0x0…0 for the root grant).

{
  "schema": "vrs/grant-chain/1.0",
  "grants": [
    { "batchId": "0x33bdafc4…", "grantor": "0x5555…5555", "grantee": "0x9999…9999",
      "quantity": "100", "parentGrant": "0x0000…0000",
      "validFrom": "1789900000", "validUntil": "1799900000", "nonce": "0x…", "signature": "0x" },
    { "batchId": "0x33bdafc4…", "grantor": "0x9999…9999", "grantee": "0x…00A1",
      "quantity": "10", "parentGrant": "0x",
      "validFrom": "1789950000", "validUntil": "1799900000", "nonce": "0x…", "signature": "0x" }
  ]
}

10. Issuer manifest — vrs/issuer-manifest/1.0

A small signed JSON object discovered at https://<domain>/.well-known/vrs/issuer.json (§8). It is not a DID document and needs no resolver. manifestHash = keccak256(JCS(manifest without manifestHash+signatures)) and may be anchored in an issuer registry.

{
  "schema": "vrs/issuer-manifest/1.0",
  "issuerRoot": "0x1111…1111",
  "roles": ["merchant"],                       // merchant|manufacturer|service|courier
  "domains": ["receipts.example"],
  "keyPolicy": { "cold": "0x1111…1111", "admin": ["0x1a..","0x1b.."] },
  "delegates": [ { "key": "0x2222…2222", "role": "merchant",
                   "validFrom": "1789990000", "validUntil": "1790600000" } ],
  "identifiers": { "legalName": "Example GmbH", "vatId": "DE123456789", "gs1CompanyPrefix": "9506000" },
  "statusEndpoint": "https://receipts.example/.well-known/vrs/status",
  "anchor": { "chainId": "8453", "registry": "0x…", "previousManifestHash": "0x…" },
  "manifestHash": "0x",
  "signatures": [ { "by": "0x1111…1111", "sig": "0x" } ]
}

11. Deployment manifest

Identifies the canonical registry deployment for an ecosystem (§13). Verifiers select registry addresses from the accepted, signed manifest; noncanonical deployments are not VRS state.

{
  "schema": "vrs/deployment-manifest/1.0",
  "ecosystem": "ethereum-base-mainnet",
  "stateProfile": "vrs/ethereum-state/1.0",
  "chainId": "8453",
  "contracts": { "entitlementRegistry": "0x…", "attestationRegistry": "0x…",
                 "anchorRegistry": "0x…", "mintAllowanceRegistry": "0x…",
                 "receiptVerifier": "0x…", "noticeRegistry": "0x…" },
  "manifestHash": "0x",
  "signatures": [ { "by": "0x", "sig": "0x…" } ]
}

12. Ref and claim responses

GET /ref/{refId} returns a powerless vrs/ref-response/1.0; claim redemption returns a one-time vrs/claim-response/1.0 carrying bearer authority (§10).

// ref-response — never contains secrets
{ "schema": "vrs/ref-response/1.0", "document": { /* signed receipt */ },
  "creditGraph": { "schema": "vrs/credit-graph/1.0", "…": "…" } }

// claim-response — returned once, marks the token consumed first
{ "schema": "vrs/claim-response/1.0", "document": { /* signed receipt */ },
  "secrets": { "schema": "vrs/secrets/1.0", "…": "…" },
  "registerAuth": [ { "lineIndex": "0", "…": "…" } ],
  "creditGraph": { "…": "…" } }

13. .vrsbox archive

The portable holder archive (§11). A ZIP/JSON container; it must not contain holder private keys unless a wallet profile permits.

{
  "schema": "vrs/vrsbox/1.0",
  "documents": [ { /* receipt */ } ],
  "secrets":   { "schema": "vrs/secrets/1.0", "…": "…" },
  "registrations": [ { "entitlementId": "0x…", "chainId": "8453", "registry": "0x…" } ],
  "lifecycle": [ { /* lifecycle attestation */ } ],
  "cache": { "creditGraphs": [], "manifests": [], "attestations": [] }
}

Field lists marked here are the canonical core; profiles may add namespaced fields under headerExtensions / lineExtensions. Where a field feeds a commitment, its type and encoding are fixed by Specification §6; this document fixes the surrounding JSON. Byte-exact values appear in the Conformance Vectors.
